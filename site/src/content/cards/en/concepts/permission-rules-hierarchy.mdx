<Band tone="paper-raised" dense>
  <h2 class="card-title">Allow/Ask/Deny Rules Hierarchy</h2>
  <p class="card-subtitle">Deny-First semantics — the first matching rule category wins; Deny is always checked first</p>
</Band>

<Band tone="tan">
  <CoreDef>Evaluation order is fixed: Deny → Ask → Allow → default. Not "highest priority wins" but "first category match wins." Even if an Allow rule appears before a Deny, Deny is still checked first. A Deny at any scope can't be overridden by any Allow.</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="eval-flow">
    <Kicker>Evaluation flow</Kicker>
    <div class="ef-step deny"><span class="ef-n">1</span><span class="ef-type">Deny?</span><span class="ef-action">match → block, stop (unoverridable)</span></div>
    <div class="ef-step ask"><span class="ef-n">2</span><span class="ef-type">Ask?</span><span class="ef-action">match → prompt for confirmation, stop</span></div>
    <div class="ef-step allow"><span class="ef-n">3</span><span class="ef-type">Allow?</span><span class="ef-action">match → permit, stop</span></div>
    <div class="ef-step fallback"><span class="ef-n">4</span><span class="ef-type">Default</span><span class="ef-action">falls back to the current Permission Mode</span></div>
  </div>
  <div class="safety-props">
    <Kicker>Three safety guarantees from Deny-First</Kicker>
    <KV label="Unoverridable across scopes">a Deny at any scope (including managed settings) can't be overridden by any Allow</KV>
    <KV label="Hooks run first">PreToolUse hook exit 2 (block) runs before rule evaluation; Allow rules can't bypass it</KV>
    <KV label="Managed Deny is unpierceable">org-level Deny is immune to --allowedTools on the command line</KV>
  </div>
  <KV label="Wildcard limits">argument-level Deny (option ordering, pipes, etc.) should prefer PreToolUse Hooks</KV>
</Band>
