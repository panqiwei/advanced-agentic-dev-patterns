<Band tone="paper-raised" dense>
  <h2 class="card-title">Fork-Explore-Commit</h2>
  <p class="card-subtitle">Fork-Explore-Commit — push git branch semantics down to the OS execution layer</p>
</Band>

<Band tone="tan">
  <CoreDef>Fork execution state at OS level into multiple exploration paths, advance each path independently, then commit the best one and discard the rest. Not code-version branching — runtime branching of agent execution state.</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="three-phases">
    <div class="phase fork-phase">
      <span class="phase-icon">⑂</span>
      <span class="phase-name">Fork</span>
      <span class="phase-desc">fork execution state at a decision point; copy-on-write gives near-zero-cost snapshots</span>
    </div>
    <div class="phase explore-phase">
      <span class="phase-icon">⊕</span>
      <span class="phase-name">Explore</span>
      <span class="phase-desc">branches advance independently; OS-level isolation keeps side effects from leaking across</span>
    </div>
    <div class="phase commit-phase">
      <span class="phase-icon">✓</span>
      <span class="phase-name">Commit</span>
      <span class="phase-desc">the chosen path's side effects become atomically effective; the rest are discarded</span>
    </div>
  </div>
  <PillarGrid cols={2}>
    <Pillar label="vs Evaluator-Optimizer">temporal exploration vs spatial exploration; complementary — each fork can internally run an eval-opt loop</Pillar>
    <Pillar label="vs Error Cascade">fork is a system-level countermeasure: errors stay in their branch, never poisoning global state</Pillar>
    <Pillar label="vs Parallelization">parallelism here means competitive selection (same problem, many paths), not task dispatch (different subtasks)</Pillar>
    <Pillar label="why OS-level">app-level checkpoints are brittle; process-level resource isolation can't be enforced by the harness</Pillar>
  </PillarGrid>
</Band>
