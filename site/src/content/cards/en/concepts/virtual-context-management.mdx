  <h2 class="card-title">Virtual Context Management</h2>
  <p class="card-subtitle">Virtual Context Management — borrow OS virtual-memory mechanics to extend the LLM's context space</p>
</Band>

<Band tone="tan">
  <CoreDef>MemGPT (Packer et al., 2023) core idea: intelligently schedule between fast storage (context window) and slow storage (external memory) to give the LLM a "virtual" context space beyond the physical window. The direct OS-LLM mapping: RAM → context window, disk → archival memory, page fault → function call (memory retrieval), LLM itself → virtual memory manager.</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="os-map">
    <Kicker>OS-LLM mapping table</Kicker>
    <div class="om-row om-head"><span>Operating system</span><span>LLM analogue</span></div>
    <div class="om-row"><span>RAM</span><span>Context window — directly visible, fast but bounded</span></div>
    <div class="om-row"><span>Disk</span><span>Archival / recall memory — large capacity, requires explicit loading</span></div>
    <div class="om-row om-key"><span>Page fault</span><span>Function call (memory retrieval) — fires when needed info is not in context</span></div>
    <div class="om-row"><span>VM manager</span><span>The LLM itself — autonomously decides when to read/write external storage</span></div>
  </div>
  <div class="strategy-unify">
    <Kicker>Unified context strategy + limits of the analogy</Kicker>
    <KV label="Compaction = in-RAM compression">Compresses data inside the context window, no disk swap involved</KV>
    <KV label="Note-taking = write-back">Persist context to external storage — an explicit write-back</KV>
    <KV label="Key difference">OS swap does not change computation results; LLM context changes directly affect reasoning quality (context rot)</KV>
    <KV label="Still necessary">Even with 1M-token context windows, context rot means hierarchical storage remains effective</KV>
  </div>
</Band>
