<Band tone="paper-raised" dense>
  <h2 class="card-title">Long-Running Agents</h2>
  <p class="card-subtitle">Long-Running Agents — shift engineers: clocking in with zero memory of what happened on the last shift</p>
</Band>

<Band tone="tan">
  <CoreDef>Task complexity exceeds a single context window; work must persist across multiple sessions. The core challenge isn't "long time" — it's that each new session starts amnesic. Context management (compaction) alone isn't enough; you need externalized, durable progress tracking.</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="failure-modes">
    <Kicker>Three failure modes</Kicker>
    <div class="fm-row"><span class="fm-type">One-shotting</span><span class="fm-desc">try to finish everything in a single session; context burns out and leaves half-done work</span></div>
    <div class="fm-row"><span class="fm-type">Premature victory</span><span class="fm-desc">see some progress and declare done — no objective evidence of completion</span></div>
    <div class="fm-row"><span class="fm-type">State fracture</span><span class="fm-desc">compaction drops critical context; next session misreads prior state</span></div>
  </div>
  <div class="solution-arch">
    <Kicker>Initializer-Coder two-agent architecture</Kicker>
    <KV label="Initializer">first run: set up environment + generate feature list (200+ items) + seed progress file</KV>
    <KV label="Coder">subsequent runs: one feature per session, keep the repo mergeable, mark passes=true only after end-to-end verification</KV>
    <KV label="State handoff">external files (progress file + git history), not the context window — immune to amnesia</KV>
  </div>
  <div class="swe-evo-note">
    <span class="sen-label">SWE-EVO EVIDENCE</span>
    <span class="sen-text">single-step 72.8% → multi-step 25%. Quantitative reason for incremental strategy — not conservatism, but necessary defense against error cascade</span>
  </div>
</Band>
