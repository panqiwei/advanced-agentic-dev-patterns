<Band tone="paper-raised" dense>
  <h2 class="card-title">Parallelization</h2>
  <p class="card-subtitle">Parallelization — multiple LLMs work on the task concurrently; outputs are merged programmatically</p>
</Band>

<Band tone="tan">
  <CoreDef>Two flavors of parallelization: Sectioning splits a task into independent subtasks running concurrently; Voting runs the same task multiple times to get diverse outputs and take consensus. Shared property: subtasks are known at design time, not decided at runtime (distinguishing it from Orchestrator-Workers).</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="two-variants">
    <div class="variant sect">
      <span class="v-title">Sectioning</span>
      <span class="v-when">fits: task splits cleanly + throughput is the bottleneck</span>
      <div class="v-examples">
        <span class="v-ex">user-query handling + content guardrail review (concurrent)</span>
        <span class="v-ex">multi-file codebase analysis in parallel</span>
      </div>
    </div>
    <div class="variant vote">
      <span class="v-title">Voting</span>
      <span class="v-when">fits: higher confidence needed + multi-angle review</span>
      <div class="v-examples">
        <span class="v-ex">multiple prompts review code for security holes from different angles</span>
        <span class="v-ex">multi-sample consensus (Self-Consistency)</span>
      </div>
    </div>
  </div>
  <div class="positioning">
    <Kicker>Place in agentic systems</Kicker>
    <KV label="vs Orchestrator-Workers">parallelization subtasks are predefined; Orchestrator-Workers decides them at runtime</KV>
    <KV label="Cost–benefit">low overhead (no extra orchestration call); suits clean-structure batch work with independent subtasks</KV>
  </div>
</Band>
