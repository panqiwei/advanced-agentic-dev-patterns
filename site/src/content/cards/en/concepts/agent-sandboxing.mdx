  <p class="card-subtitle">Agent Sandboxing — isolation for the age of dynamic code generation</p>
</Band>

<Band tone="tan">
  <CoreDef>Classical sandboxes assume code is static and known. Agents face dynamic code generation, semantic prompt injection, permission propagation across tool chains, and context contamination — four new attack surfaces classical sandboxes can't cover.</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="approach-stack">
    <div class="approach a-exec">
      <span class="approach-name">Execute-Only Agents</span>
      <span class="approach-tag">Architectural prompt-injection defense</span>
      <p>Strictly separate planning from execution — the execution layer only runs a pre-approved action set, has no language understanding, and so removes the injection surface at the root</p>
    </div>
    <div class="approach a-grimlock">
      <span class="approach-name">Grimlock</span>
      <span class="approach-tag">eBPF + trusted channel</span>
      <p>Kernel-level behavior monitoring + enforced policy; one framework delivers both observability (monitor) and security (block)</p>
    </div>
    <div class="approach a-dynamic">
      <span class="approach-name">Dynamic sandbox</span>
      <span class="approach-tag">Lightweight runtime</span>
      <p>Ultra-low-overhead sandbox create/destroy for agent-generated code; an agent may spawn dozens of snippets per minute</p>
    </div>
  </div>
</Band>
