<Band tone="paper-raised" dense>
  <h2 class="card-title">分叉-探索-提交</h2>
  <p class="card-subtitle">Fork-Explore-Commit — 将 git 分支语义下沉到 OS 执行层</p>
</Band>

<Band tone="tan">
  <CoreDef>在 OS 级 fork 执行状态为多条探索路径，各路径独立推进，最终选择最优路径 commit，丢弃其余。不是代码版本控制的分支——是运行时 agent 执行状态的分支。</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="three-phases">
    <div class="phase fork-phase">
      <span class="phase-icon">⑂</span>
      <span class="phase-name">Fork</span>
      <span class="phase-desc">在决策点 fork 执行状态；copy-on-write 实现近零开销快照</span>
    </div>
    <div class="phase explore-phase">
      <span class="phase-icon">⊕</span>
      <span class="phase-name">Explore</span>
      <span class="phase-desc">各分支独立推进；OS 级隔离，副作用不跨分支泄漏</span>
    </div>
    <div class="phase commit-phase">
      <span class="phase-icon">✓</span>
      <span class="phase-name">Commit</span>
      <span class="phase-desc">选中路径的副作用原子性生效；其余分支丢弃</span>
    </div>
  </div>
  <PillarGrid cols={2}>
    <Pillar label="vs Evaluator-Optimizer">时间维度探索 vs 空间维度探索；互补——每条 fork 内可运行 eval-opt 循环</Pillar>
    <Pillar label="vs Error Cascade">Fork 是系统级对抗：错误只影响所在分支，不污染全局状态</Pillar>
    <Pillar label="vs Parallelization">并行是为了竞争择优（同问题多路径），不是任务分发（不同子任务）</Pillar>
    <Pillar label="为什么需要 OS 级">应用层 checkpoint 脆弱；进程级资源隔离无法被 harness 强制</Pillar>
  </PillarGrid>
</Band>
