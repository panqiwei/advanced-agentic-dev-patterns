<Band tone="paper-raised" dense>
  <h2 class="card-title">长时运行 Agent</h2>
  <p class="card-subtitle">Long-Running Agents — 轮班工程师：上班时对前一班发生的事毫无记忆</p>
</Band>

<Band tone="tan">
  <CoreDef>任务复杂度超出单个 context window，需要跨多 session 持续工作。核心挑战不是"时间长"，是每个新 session 天然失忆。仅靠 context management（compaction）不足——需要外部化的持久进度追踪。</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="failure-modes">
    <Kicker>三类失败模式</Kicker>
    <div class="fm-row"><span class="fm-type">One-shotting</span><span class="fm-desc">试图在单 session 内完成全部工作，context 耗尽留下半成品</span></div>
    <div class="fm-row"><span class="fm-type">Premature victory</span><span class="fm-desc">看到已有进展过早宣布完成——没有客观的完成度证据</span></div>
    <div class="fm-row"><span class="fm-type">状态断裂</span><span class="fm-desc">compaction 丢失关键上下文，下一 session 误解前序状态</span></div>
  </div>
  <div class="solution-arch">
    <Kicker>Initializer-Coder 双 Agent 架构</Kicker>
    <KV label="Initializer">首次运行：环境搭建 + feature list 生成（200+ 条）+ 进度文件初始化</KV>
    <KV label="Coder">后续运行：每次只做一个 feature，保持代码库可合并，端到端验证后标记 passes=true</KV>
    <KV label="状态传递">外部文件（progress file + git history）而非 context window——防止失忆</KV>
  </div>
  <div class="swe-evo-note">
    <span class="sen-label">SWE-EVO 证据</span>
    <span class="sen-text">单步 72.8% → 多步 25%。增量推进策略的量化理由——不是保守，是在误差级联面前的必要手段</span>
  </div>
</Band>
