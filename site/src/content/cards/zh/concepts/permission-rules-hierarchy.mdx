<Band tone="paper-raised" dense>
  <h2 class="card-title">Allow/Ask/Deny 规则层次</h2>
  <p class="card-subtitle">Deny-First 语义 — 第一个匹配的规则类别胜出，Deny 始终最先检查</p>
</Band>

<Band tone="tan">
  <CoreDef>评估顺序固定：Deny → Ask → Allow → 默认。不是"最高优先级 wins"，而是"第一个类别匹配 wins"。即使 Allow 规则排在 Deny 之前，Deny 仍先被检查。跨层级 Deny 无法被任何 Allow 覆盖。</CoreDef>
</Band>

<Band tone="paper-raised" grow>
  <div class="eval-flow">
    <Kicker>评估流程</Kicker>
    <div class="ef-step deny"><span class="ef-n">1</span><span class="ef-type">Deny?</span><span class="ef-action">匹配 → 阻止，终止（不可被覆盖）</span></div>
    <div class="ef-step ask"><span class="ef-n">2</span><span class="ef-type">Ask?</span><span class="ef-action">匹配 → 弹出确认提示，终止</span></div>
    <div class="ef-step allow"><span class="ef-n">3</span><span class="ef-type">Allow?</span><span class="ef-action">匹配 → 直接允许，终止</span></div>
    <div class="ef-step fallback"><span class="ef-n">4</span><span class="ef-type">默认</span><span class="ef-action">取决于当前 Permission Mode</span></div>
  </div>
  <div class="safety-props">
    <Kicker>Deny-First 的三项安全保证</Kicker>
    <KV label="跨层级不可覆盖">任何作用域（包括托管设置）的 Deny 无法被其他层的 Allow 覆盖</KV>
    <KV label="Hooks 优先">PreToolUse hook exit 2（阻止）在规则评估之前运行，Allow 规则不能绕过</KV>
    <KV label="托管 Deny 不可穿透">组织 Deny 不受命令行 --allowedTools 影响</KV>
  </div>
  <KV label="通配符局限">复杂参数级 Deny（如选项顺序变化、管道）应优先使用 PreToolUse Hooks 实现</KV>
</Band>
